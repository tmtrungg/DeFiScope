A guided reading of the DeFiScope artifact · ASE 2025

DeFiScope, annotated.

How a research tool uses a large language model to catch DeFi price-manipulation attacks — explained from zero, using the tool's own code and one real transaction it was run on.

You already know a little DeFi. You know nothing about LLMs yet. By the end you'll know what fine-tuning, prompts, tokens, and chain-of-thought actually are — not as abstractions, but as specific lines in this repository — and you'll be able to build something like it yourself.

Paper: Zhong, Wu, Liu et al., "Detecting Various DeFi Price Manipulations with LLM Reasoning", IEEE/ACM ASE 2025.
Code: github.com/AIS2Lab/DeFiScope  ·  This guide pairs paper sections (§) with real files (utils/…).
Everything quoted here is verbatim — the prompts were produced by actually running the fixed tool on the AES attack.

Why this tool exists

01The problem: prices you can shove around

A one-screen refresher, because the whole design follows from it.

In DeFi there is no order book and no exchange setting the price. A pool holds two token reserves, and an equation turns those reserves into a price. The classic one is the Constant Product Market Maker:

x · y = k  →  instantaneous price of token x, in units of y, is P = y / x.

So if you dump a huge amount of token x into the pool, x shoots up, and the pool's own formula now reports token x as cheaper. That is not a bug — it is how the price is defined. An attacker exploits it in three beats: distort a pool's balances, do something valuable while the reported price is wrong (borrow, redeem, sell into a second pool), then unwind. Flash loans make the capital free.

The gap in prior tools

State-of-the-art detectors (DeFiRanger, DeFort) work by computing the exchange rate between two tokens and watching it move. That only works when the price follows a standard formula they can compute. But 44.2% of the 95 attacks the authors collected used non-standard, custom price models — a median-of-oracles, an exponential curve, a lending formula reading multiple balances. For those, "the exchange rate" isn't even well-defined. The paper's motivating case, UwULend ($19.3M), moved sUSDe's price by only ~4%, well inside DeFort's "normal" band, and hid the manipulation inside complex WETH deposits that DeFiRanger couldn't trace.

The authors' insight: you don't need to compute the price. You only need to know which direction it moved. And direction can be read off two things a human analyst would use — the pricing code and the balance changes — without any protocol-specific math. That "read the code, reason about direction" step is exactly what a language model is good at and ordinary program analysis is bad at. Hence: bring in an LLM.

The architecture in one sentence

02The LLM is a sensor. Plain code is the judge.

This split is the most important thing to understand about the whole system — and about most well-built LLM tools.

The model never decides "attack or not". It answers one narrow perception question, and deterministic code does everything else.

The LLM is asked exactly one kind of thing, over and over: "Given this pricing code and these balance changes, did token T's price go up or down?" That's it. Recovering who did what, linking the steps, matching attack shapes, producing the verdict — all of that is ordinary Python you could step through in a debugger. Look at how little surface the model actually touches:

1
Fetch & decode the transactiontransaction.pyOne RPC call (debug_traceTransaction) returns the whole nested call tree with event logs. Decode calldata & Transfer/Mint/Burn events into structured records.
2
Slice into "user invocations"tranxToUserCalls.pyCut the tree wherever the attacker's contract starts a new external call. Each slice is one unit of reasoning.
3
Build the Transfer GraphtransferGraph.pyAccounts are nodes; token transfers are timestamped directed edges. Pure graph theory.
4
Recover DeFi operationsdefiAction.pySearch the graph for Swap / Deposit / Withdraw / Borrow / Stake / Claim. E.g. a Swap is a cycle of transfers that leaves and returns to the user with a different token.
5
Extract the pricing codefunction.pyKeyword-match function names (price, getAmount, …), download verified source, use Slither to pull the relevant functions' text.
6
Build the promptpriceChangeInference.pyGlue the code + balance-change sentences + a strict answer template into one text string.
7
Ask the fine-tuned LLM← the only learned stepThe model extracts the price model and scores four statements ("price up / price down", per token) from 1–10. This is the entire footprint of machine learning in the tool.
8
Turn scores into a directionpriceChangeInference.pyHigher of the up/down pair wins → INCREASE / DECREASE / UNCERTAIN. A one-line argmax, no threshold.
9
Match against 8 attack patternsdetector.pyHand-written rules: "operation A, then a manipulated price in pool P, then operation C" → fire.
Why you should care about this split

When the paper reports 96% precision and zero false alarms on 96,800 benign transactions, most of that reliability comes from the deterministic gates, not the model. The fuzzy component is boxed into one well-defined sub-question with a machine-checkable output. That is the reusable lesson: use the LLM only where code can't go, and make its job as small and as checkable as possible.

The clever data structure underneath: the Transfer Graph

Steps 3–4 deserve a closer look, because recovering "who did what" is the paper's second contribution (§V) and the reason it beats the previous best tool. Every token transfer becomes a timestamped directed edge — sender → receiver — and accounts are nodes. Then each DeFi operation is just a graph shape:

  • Swap = a cycle that leaves a user account and returns with a different token, along edges of strictly increasing time.
  • Deposit = a transfer into a pool plus a mint of a receipt token back to the user.
  • Borrow = a mint of a debt token + assets flowing out, with nothing flowing in.

DeFiRanger, the prior state of the art, used a "Cash Flow Tree" that could only join transfers touching the user directly — so a swap routed through several relayer contracts (U → A → B → C → U) broke it. The Transfer Graph treats relayers as ordinary intermediate nodes. On the authors' 1,000-transaction test this lifted operation-recovery recall from 0.559 to 0.912, and much of that gap is exactly these multi-relayer swaps. This is the "who did what" half of detection — entirely deterministic, no model involved.

The vocabulary, grounded in real lines

03Everything you need to know about LLMs, taught through this repo

Each idea below is defined once, then pinned to the exact place it appears in DeFiScope. Nothing here is abstract.

First: what a large language model actually is

The one-paragraph mental model, before anything else. An LLM is a next-token predictor: you give it some text, it outputs a probability for every possible next token, picks one, appends it, and repeats. That single loop is the whole mechanism — "reasoning", "answering", "writing Solidity" are all just very good next-token prediction.

Where does the skill come from? Pretraining. Before anyone fine-tunes anything, the model is trained on an enormous slice of the internet — books, code, docs — by playing "guess the next token" billions of times. That is how GPT-3.5 and Phi-3 already know English, already know Solidity syntax, and already know the x·y=k rule: nobody in this project taught them that, it came free from pretraining. A model fresh out of pretraining is called a base model.

DeFiScope's problem is that a base model knows the concepts but doesn't reliably apply them in the rigid format the tool can parse. Closing that gap is exactly what fine-tuning (§5) does. Hold this frame as you read the rest: everything is either feeding the predictor better input text (a prompt) or nudging what it tends to predict (fine-tuning).

A "token" is the unit of text the model reads and is billed by

An LLM doesn't see characters or words — it sees tokens, sub-word chunks (roughly ¾ of an English word each). Two consequences run through the whole system. First, every model has a context window: a hard cap on tokens per request. Second, API pricing is per token. That's why the pipeline never pastes a raw transaction (which would be hundreds of thousands of tokens) into the prompt — it compresses everything to a few sentences. You can see the window arithmetic literally spelled out for the local model:

utils/gen_with_local_model.pypythonnew_tokens = 128000 - prompt_length  # Phi-3's 128k-token window minus the prompt
if new_tokens <= 0:
    raise ValueError("The prompt length exceeds or equals 128k tokens!")

The paper's headline cost, $0.0107 per transaction, is just (prompt tokens + answer tokens) × the per-token price. Prompt size is engineered, not accidental.

A "prompt" is just a string your program builds

There is no magic API for "give the model some code and some numbers". You concatenate everything you want it to know into one text string. Here is the actual template DeFiScope wraps around a closed-source pool — read it as ordinary string-building:

utils/priceChangeInference.py  ·  the "Type-II" promptpythoninstruction = """
{pool_address} is the address of a liquidity pool. The price model of
the pool aligns with the Constant Product Market Maker (CPMM). You will
be provided with some changes of tokens' balance inside the pool. Only
based on the given information, ... give me evaluation scores from 1 to
10: {statements}. There is no need for quantitative calculation."""

Two prompt shapes exist, and the routing between them is plain Python, not the model's choice:

Type-I · open source

Verified pricing code was found. Paste the actual Solidity functions and ask the model to extract the price model from them, then score. Best grounding.

Type-II · closed source

No code, but the graph proved it's a 2-token pool. Just assert "this is a CPMM" and let the model apply the x·y=k rule it already learned in pretraining.

"System" vs "user" messages

Chat models take a list of messages with roles. A system message sets standing behavior; the user message carries the task. DeFiScope's system message is a one-liner that puts the model in character:

utils/priceChangeInference.pypythonmessages=[
  {"role": "system", "content": "You are a price oracle of DeFi protocols,
     your job is to evaluate the price change of assets..."},
  {"role": "user",   "content": prompt}   # the string built above
]

"Temperature" is the randomness dial

At each step the model produces a probability distribution over the next token. Temperature rescales it before sampling. At temperature=0 the model always takes the single most likely token, so the same input gives the same output every time — essential for a security tool whose verdict must be reproducible.

utils/priceChangeInference.pypythontemperature = 0      # deterministic: always the likeliest token
top_p       = 1      # no nucleus filtering

The local model uses temperature=1e-8 instead of exactly 0 — a quirk, because HuggingFace's sampler rejects a literal 0. Same intent.

A "fine-tuned model" is a private, specialized copy — and it's one string

We'll go deep in §6. For now, know that this single line is where all the machine learning enters, and it's why the repo can't be run as-is by anyone else:

utils/priceChangeInference.py  ·  original line 233pythonmodel = "ft:gpt-3.5-turbo-1106:metatrust-labs::8zFctmxs"  # fine-tuned

Decode the string: ft: = fine-tuned, gpt-3.5-turbo-1106 = the base model, metatrust-labs = the owning organization. That model lives inside the authors' OpenAI account; your key gets "model not found". (Our fixed copy reads it from an environment variable instead — see §8.)

The failure mode every LLM engineer must internalize

LLM output is untyped text, and API calls fail. DeFiScope catches any API error and substitutes zero scores → every direction becomes UNCERTAIN → the verdict becomes "not an attack". So a wrong key or an inaccessible model makes the tool confidently report every attack as benign, with no crash. When you evaluate any LLM tool, first prove the model calls actually succeed before you trust a single recall number.

Now the reasoning the fine-tuned model is trained to do: AES balance increased massively, so by x·y=k there's more AES in the pool → AES is cheaper → statement 2 ("AES decreases") scores high (say 8), statement 1 low (say 2). A regex lifts the integers out of the reply, and this converts them to a direction — note there is no absolute threshold, just which of the pair is larger:

utils/priceChangeInference.py  ·  generate_finally_predictionpythonif   increase_score > decrease_score: tendency = Tendency.INCREASE
elif increase_score < decrease_score: tendency = Tendency.DECREASE
else:                                tendency = Tendency.UNCERTAIN

Finally the deterministic judge. DeFiScope tests ordered triples of user calls — (buy, manipulate, sell) — and fires only if the middle call carried a manipulated price in the same pool the outer calls touch. For AES: buy AES cheap, having pushed its price down, then sell into the mispriced pool. That is Pattern I, and it's a hand-written rule in detector.py, not anything the model concluded.

What "no key" proves

Because steps 1–9 minus the model are pure code, the no-key run exercised the entire program-analysis half of the paper — trace decoding, graph construction, Swap recovery on a real pool, prompt generation — and emitted six well-formed prompts. The only thing missing is the model's up/down call. That's the cleanest possible demonstration of the sensor/judge split.

The heart of the contribution  ·  paper §IV

05Fine-tuning, explained from scratch

This is the part the paper is proudest of, and the part a newcomer most needs demystified. We'll define it, show the actual training data, and explain why it's novel.

What fine-tuning is

A pretrained model (GPT-3.5, Phi-3) already knows English and some Solidity. Fine-tuning means continuing its training on a small set of your own (input, desired-output) examples, so that its default behavior on your task matches those examples. Because every input ships with the correct output, this is called Supervised Fine-Tuning (SFT). In this repo, one example is literally one row of a CSV:

dataset/training_set.csv  ·  row 1 — the "input" column▲ TRAINING DATA · verbatim
Instruction 1: The following is related price calculation functions. You are required to extract the price calculation model. function swapExactTokensForTokens(uint amountIn, ... ) { amounts = UniswapV2Library.getAmountsOut(factory, amountIn, path); ... } function getAmountOut(uint amountIn, uint reserveIn, uint reserveOut) { // the x*y=k constant-product formula, with the 997/1000 fee ... } Instruction 2: ... evaluate ... scores from 1 to 10: 1) input token price increases 2) input token price decreases 3) output token price increases 4) output token price decreases <change> input balance increases by ...; output balance decreases by ... </change>

"Chain-of-Thought" — the trick that makes it generalize

Look at the paired response the model is trained to produce. It does not jump to four numbers. It reasons first, then answers:

dataset/training_set.csv  ·  row 1 — the "response" column▲ THE TARGET · verbatim, abridged
To evaluate the statements ... we analyze how the changes affect the reserves used in the getAmountOut function. 1. The balance of input token increases, meaning its reserve increases. 2. The balance of output token decreases, meaning its reserve decreases. An increase in the reserve of the input token generally leads to a decrease in its price relative to the output token ... 1) input price increases → Evaluation Score: 2 2) input price decreases → Evaluation Score: 8 ...

Training on the reasoning, not just the final scores, is what "CoT-style fine-tuning" means. It teaches the model the derivation ("more reserve → cheaper"), which is a general principle — and that is why a model trained only on Uniswap-V2 CPMM data can later score custom price models it never saw.

How the model learns only the answer

A subtle but essential mechanic. During training you show the model prompt + response concatenated, but you only want it to learn to produce the response. That's done by masking the prompt's tokens with the sentinel -100, which PyTorch's loss function ignores:

fine-tuning/fine-tuning.pypythonlabels["input_ids"][i] = [-100] * len(prompt_tokens) + response_tokens
#                        ^ ignore the prompt      ^ learn to produce this

So gradient descent only ever rewards reproducing the desired answer given the prompt — never parroting the question back. This one line is the definition of instruction SFT in practice.

Two ways to fine-tune — the repo shows both

● OpenAI-hosted (the paper's main results)

You upload the examples; OpenAI trains on their servers; you get back an opaque model id and nothing else. No weights, no control. That id — ft:gpt-3.5-turbo-1106:metatrust-labs::… — is all that survives in the repo, usable only by the authors' key. This is the single biggest reproducibility gap.

● Local LoRA (shipped & runnable)

fine-tuning/fine-tuning.py fine-tunes an open Phi-3 model on your own hardware, and the adapter weights are yours to keep and share.

LoRA: cheap fine-tuning, in one picture

Phi-3-medium has ~14 billion parameters; updating all of them needs enormous memory. LoRA (Low-Rank Adaptation) freezes the entire base model and trains tiny "adapter" matrices bolted onto a few layers — a fraction of a percent of the weights:

fine-tuning/fine-tuning.pypythonlora_config = LoraConfig(
    r=4,                              # adapter rank — tiny
    lora_alpha=8,
    target_modules=["o_proj", "qkv_proj"],  # only the attention layers
    task_type=TaskType.CAUSAL_LM,
)

That's why the released artifact is a few-MB adapter, and why loading it (utils/load_model.py) first downloads the full base model and then layers the adapter on top. LoRA is the reason "fine-tune your own 14B model" is feasible on a handful of GPUs instead of a datacenter.

Why the fine-tuning matters — the money result

The ablation is the paper's strongest evidence. Fine-tuning on CPMM data only lifted detection of attacks on custom price models from 60% to 93.3% for GPT-3.5 — a model that never saw a custom formula got dramatically better at them. That's transfer: CoT taught transferable "reserve-ratio" reasoning, not memorized answers. Overall, fine-tuning raised detection by up to 31% over the same model un-tuned. The base model already knows what a price is; fine-tuning makes it reliably answer in the exact scored format the downstream code can parse, every time.

Standard ML discipline you can point to

The 1,000 synthesized samples are split 800 / 100 / 100 (train / eval / test). Train on 800; after each pass, measure loss on the 100 eval samples to catch overfitting; report final quality on the 100 test samples the training never touched. The labels themselves came from a program, not humans: Foundry fuzzed real swaps against a Uniswap-V2 pool, so x·y=k gives the ground-truth direction for free (500 "inflate" + 500 "deflate" pairs).

06The eight attack patterns

A price wobble alone is meaningless — a normal swap moves prices too. It becomes an attack only inside a shape.

DeFiScope enumerates ordered triples of user calls and checks each against eight hard-coded patterns, grouped into four attack types. In every one, the middle (or first) step is where the LLM said a price was manipulated, and the outer steps are the profit:

TypeShapeThe idea, in wordsCode
Buy & Sell
I, II		buy → manip → sellPush a token's price in one pool, then trade it favorably in another.detector.py:72
Deposit & Borrow
III, IV		deposit → manip → borrowInflate your collateral's reported value, then borrow more than it's worth. (Cream Finance, $130M.)detector.py:217
Stake & Claim
V, VI		stake → manip → claimDistort the reward-pricing, then claim inflated rewards.detector.py:497
Deposit & Withdraw
VII, VIII		deposit → manip → withdrawDeposit for a receipt token, move the price, redeem the receipt for more than you put in. (Harvest, $21.5M.)detector.py:362

The two patterns per type differ only in when the manipulation happens (before vs. between the profit steps). The key efficiency: the expensive LLM calls happen once per user call, up front; the triple-loop then reuses those cached directions with zero extra model calls — which is how a cubic combination count still runs in ~2.5s per transaction.

How they prove it works  ·  paper §VII

07How the paper backs its claims — and how you'd evaluate your own tool

For research, the evaluation design matters as much as the method. This is the part you will most directly reuse.

Two words first. Recall = of all the real attacks, what fraction did the tool catch? (Did it miss things?) Precision = of everything the tool flagged, what fraction were truly attacks? (Did it cry wolf?) A detector needs both high — you can get 100% recall trivially by flagging everything, at ruinous precision.

Three datasets, three questions

DatasetWhat it isQuestion it answers
D1 — 95 attacksReal, confirmed price-manipulation hacks (pooled from DeFort, the DeFiHackLabs repo, and an industry partner)Recall — how many real attacks get caught
D2 — 968 suspiciousProfitable-but-mixed transactions, deliberately including other attack types (reentrancy, access control…)Precision — does it flag only price manipulation
D3 — 96,800 benignEveryday transactions from top DeFi apps, at a realistic 1:100 suspicious-to-benign ratioFalse alarms on normal traffic

The headline numbers, and what they mean

  • 80% recall on D1 — caught 76 of 95, vs 52.6% (DeFort), 51.6% (DeFiRanger), 35.8% (DeFiTainter). Those three are the prior state of the art; beating all of them is the core result (RQ1).
  • 96% precision on D2 — 147 of 153 flags were genuine, and 81 were previously-undocumented incidents.
  • Zero false alarms on 96,800 benign transactions — the practicality result (RQ3), at ~2.5s and ~$0.01 per transaction.

The ablation — the experiment that isolates the idea

An ablation removes one ingredient to measure its contribution. Here: run the identical pipeline with the model un-fine-tuned vs fine-tuned. This is the single most important table for you, because it's the evidence that fine-tuning — not merely "using an LLM" — is what does the work:

Read the middle row against the top: fine-tuning on CPMM-only data pushed custom-price-model detection from 60% → 93.3% — the transfer-learning claim, in one number. And cheap fine-tuned GPT-3.5 matches expensive fine-tuned GPT-4o, which is why they ship the cheap one. They confirm the gains are statistically significant with a McNemar test (p < 0.05) — the standard test for comparing two classifiers on the same set of cases.

Limitations the paper admits — your research openings

DeFiScope misses cross-transaction attacks (it analyses one transaction at a time; 8 of the 11 "detectable but missed" cases were multi-transaction), struggles when pricing needs heavy arithmetic (LLMs are weak at exact computation — a suggested fix is letting the model write and run a script), and needs source code (closed-source contracts fall back to the CPMM assumption). Each of these is a legitimate "next paper" if you want to build on it.

Answering your real question

08Did the authors hide something? What's real, what's missing

You suspected the materials might not run or might be incomplete. Here's the honest audit — verified by actually running it on 2026-07-05.

Short version: the deterministic pipeline genuinely works and the datasets check out, but the paper's headline LLM results are not reproducible from the repo alone — not from malice, but because the crucial learned artifact lives on someone else's server and the data-synthesis code was never shipped. Two categories:

Genuinely missing artifacts (not fixable by editing code)

  • The OpenAI fine-tuning recipe — no CSV→JSONL converter, no job script; and gpt-3.5-turbo-1106 fine-tuning has since been retired, so the exact model can never be rebuilt. You can re-fine-tune a current base model from the shipped CSV (recipe in REPRODUCTION.md).
  • The Foundry data-synthesis pipeline — only the finished training CSVs are shipped, not the fuzzer that made them.
  • fine-tuning/lora_ds_config.json — referenced by the training script but absent; fine-tuning crashes without it. (Its contents are printed in model_comparison_result.md.)
  • Ground-truth labels for D2 / D3 / the TG-vs-CFT experiment — so the 96% precision and TPR 0.912 can't be recomputed from the repo.
  • No batch harness — main.py takes one transaction; nothing loops over the datasets.

None of this contradicts the paper's claims — it just means "code + data on GitHub" is not the same as "end-to-end reproducible", a distinction worth carrying into your own RQ3 work.

For your own research

09If you want to build something like this

The transferable recipe, distilled from what DeFiScope does well.

  1. Find the one sub-problem code can't solve. Here it's "read arbitrary pricing Solidity and infer price direction." Everything else stayed deterministic. Your LLM footprint should be embarrassingly small.
  2. Constrain the output to something machine-checkable. Four 1–10 scores on paired opposing statements — not free text. The pairing is a built-in consistency check, and the argmax has no fragile threshold.
  3. Bias failure toward safe. Any uncertainty → UNCERTAIN → "no attack." On a stream that's 99% benign, you want misfires to be false negatives, not false alarms.
  4. Fine-tune for format reliability and transferable reasoning. Use CoT targets so the model learns the derivation; generate labels programmatically (fuzzing gave free ground truth) so you're not hand-labeling.
  5. Compress the input. Don't paste raw data; distill it to the few facts the model needs. This controls both cost and the context window.
  6. Log the raw model output and treat every call as fallible. The one anti-pattern to avoid: silently converting API errors to a neutral answer. Distinguish "model said no" from "model never answered."
Try it yourself, free

In your fixed clone: unset OPENAI_API_KEY && python main.py -tx 0xca4d0d24aa448329b7d4eb81be653224a59e7b081fc7a1c9aad59c5a38d0ae19 -bp bsc, then read prompts_dump/. You'll watch the deterministic pipeline turn a raw hash into the exact questions the LLM would be asked — the best way to feel the architecture in your hands.

10Pocket glossary

Token

Sub-word chunk of text (~¾ word) the model reads; the billing unit. gen_with_local_model.py

Context window

Max tokens per request. Phi-3-medium-128k → 128,000.

Prompt

The single text string you send. Built by string concatenation. priceChangeInference.py

System / user message

Standing instruction vs. the per-request task, sent as roles.

Temperature

Randomness dial. 0 = deterministic (what a detector needs).

Fine-tuning (SFT)

Continue training a base model on your own (input, output) pairs so it defaults to your task.

Chain-of-Thought (CoT)

Training/asking the model to reason step by step before answering. Improves and generalizes reasoning.

LoRA

Freeze the big model, train tiny adapter matrices on a few layers. Cheap, portable fine-tuning. r=4

Label masking (-100)

Ignore prompt tokens in the loss so the model learns only to produce the answer.

Base vs. fine-tuned model

gpt-3.5-turbo vs. ft:gpt-3.5-turbo-…:org::id — the latter is a private specialized copy.

Sensor / judge split

LLM (next-token predictor)

A model that, given text, repeatedly predicts the next token. All its abilities are that loop, done well.

Pretraining

The first, huge training pass on internet text. Where the model learned English, Solidity, and x·y=k for free.

Recall / precision

Recall = fraction of real attacks caught. Precision = fraction of flags that are real. A detector needs both.

Ablation

Remove one ingredient (e.g. fine-tuning) to measure exactly what it contributes.

Hallucination

An LLM stating something false with full confidence. Why its output is untrusted text you must parse and check.
